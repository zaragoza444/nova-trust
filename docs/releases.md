# Releases

This repository uses a simple tag-driven release flow.

## Versioning

Use semantic version tags:

- `v0.1.0`
- `v0.2.0`
- `v1.0.0`

## Before Releasing

1. Make sure `main` contains the changes you want to ship.
2. Update `CHANGELOG.md` with the release summary.
3. Run the release validation locally:

```bash
npm install
npm run release:check
```

## Create a Release

From `main`:

```bash
git checkout main
git pull origin main
git tag -a v0.1.0 -m "Release v0.1.0"
git push origin v0.1.0
```

Pushing a tag that matches `v*.*.*` triggers the GitHub Actions workflow in `.github/workflows/release.yml`.

That workflow will:

1. install dependencies
2. run `npm run release:check`
3. create a GitHub Release with autogenerated notes

## Release Contents

The initial workflow publishes release notes and the release tag. It does not yet attach compiled binaries or container images.

## Rollback

If a release is bad:

1. revert or fix the issue on a new branch
2. merge the fix into `main`
3. cut a new patch release, for example `v0.1.1`

Avoid deleting release tags that may already be referenced externally.
